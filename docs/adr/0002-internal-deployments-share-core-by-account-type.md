# 内部部署通过账号类型复用共享核心

PreviewShip 的公司内部部署与普通用户部署共用 `users`、`projects`、`deployments`、标准 `/v1` API 和部署 Worker，以 `users.account_type` 作为唯一业务身份来源，并由账号策略统一决定额度、过期、保留、品牌、统计、事件和通知行为。我们拒绝独立内部表与独立部署接口，因为它们会复制项目去重、固定链接、产物处理、并发激活、回滚和安全校验；也拒绝按邮箱、API Key 或请求参数硬编码，因为同一账号还需要通过 Session 登录维护，身份必须在两种鉴权方式下保持一致。

## Consequences

- 普通账号默认保持 `CUSTOMER`，内部身份由受控运维操作授予且不恢复为普通账号。
- 内部部署只做业务隔离，不做队列、Worker、临时上传目录或产物存储的物理隔离。
- 所有经营查询必须排除内部账号；技术可观测性和 Abuse 安全能力仍保留。
- 内部回滚创建新的部署记录并关联来源版本，历史部署记录保持不可变。
