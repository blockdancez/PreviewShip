# Pro 套餐项目数调整设计

## 背景

后端 `application.yml` 已由用户将 `PRO_MONTHLY` 最大项目数调整为 20，将 `PRO_YEARLY` 最大项目数调整为 40。其余展示文案、公开 SEO 内容和测试仍存在旧的 10/20 配置，需要保持全系统一致。

## 目标

- `PRO_MONTHLY` 最大项目数统一为 20。
- `PRO_YEARLY` 最大项目数统一为 40。
- 保留后端 YAML 中用户已经完成的修改，不重复编辑。
- 不影响 Free 套餐、价格、部署次数、上传大小、流量额度或部署版本保留数。

## 实施范围

### 前端与多语言

更新套餐比较数据以及 8 种语言的落地页和文档文案，使所有面向用户的项目数说明统一为 20/40。

### SEO 与公开文档

更新公开 SEO 文件的生成源，并重新生成对应产物，避免搜索引擎和 LLM 文档继续展示旧额度。

### 后端测试

更新权益服务和部署服务测试中的套餐额度边界，使测试覆盖月付 20、年付 40 的真实配置。

### 项目文档

更新产品和运维文档中明确表示 Pro 项目容量的旧数字。与项目容量无关的数字保持不变。

## 数据流与兼容性

运行时权益仍由后端 YAML 经 `EntitlementsService` 提供，接口结构和错误处理不变。前端套餐数据和静态文档是展示副本，本次只同步数值，不新增接口、数据库迁移或缓存逻辑。

## 验证

- 后端运行完整 Maven 测试。
- 前端运行 lint、单元测试和生产构建。
- 全局检索旧的项目额度表述，逐项确认剩余的 10/20 是否属于版本保留数、部署次数等其他业务含义。
- 检查 Git 差异，确保后端 YAML 保持为用户修改，且无无关文件被覆盖。
