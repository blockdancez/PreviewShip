# PreviewShip 管理员增长经营驾驶舱设计

## 背景与目标

PreviewShip 已经拥有产品事件、用户、项目、部署、结账、订阅和支付数据，也已有通过 `PREVIEWSHIP_ADMIN_EMAILS` 判断管理员的权限链路。当前缺少的不是更多原始报表，而是一张帮助管理员快速判断趋势、解释转化阻塞并决定下一步行动的经营视图。

新页面命名为“增长经营”，路径为 `/admin/analytics`。它必须在十秒内回答：

1. 最近一个周期业务变好还是变差；
2. 用户旅程在哪一步出现主要流失；
3. 哪些来源、地区、页面、部署渠道或失败原因造成变化；
4. 管理员现在最值得执行什么动作。

第一版只读取 PreviewShip 自有数据库，不在页面运行时调用 GA、Google Ads、Google Search Console 或 Bing API。

## 已确认的设计原则

- 页面是增长经营驾驶舱，不是原始埋点列表或自由拖拽 BI 工具。
- 增长与转化为主，稳定性和风控只作为异常提醒及下钻入口。
- 漏斗以唯一用户旅程为主口径，事件次数作为辅助口径。
- 经营建议由确定性规则生成，必须附带指标、基线、影响范围和行动。
- 默认统计最近七个完整自然日，并与前七日比较。
- 自然日统一使用 `Asia/Shanghai`，支持今天、七天、三十天、九十天和自定义日期。
- 7 日激活只统计已经拥有完整观察窗口的 cohort。
- UI 采用“经营叙事”结构：结论、结果、旅程、原因、行动依次展开。
- 正式实现必须兼顾视觉美感、可操作性、无障碍和响应式体验。

## 权限与入口

- 继续使用 `PREVIEWSHIP_ADMIN_EMAILS → AdminService → /me.isAdmin` 权限机制，不新增角色表。
- 侧边栏仅对管理员显示“增长经营”入口。
- 前端隐藏入口只是体验措施；后端每个分析接口都调用 `AdminService.requireAdmin(userId)`。
- 非管理员访问接口返回 403，直接访问页面时显示无权限状态，不泄露任何汇总值。

## 后端架构

新增以下职责单一的组件：

- `AdminAnalyticsController`：解析和校验时间范围，执行管理员校验，返回统一响应。
- `AdminAnalyticsService`：组织统计周期、当前与对比周期、聚合查询及响应模型。
- `AdminAnalyticsRepository`：使用 PostgreSQL 聚合查询计算趋势、唯一旅程、来源和诊断维度，不加载原始事件集合。
- `AdminAnalyticsInsightService`：根据聚合指标生成结构化经营结论和建议。

第一版提供单一读接口：

```http
GET /console/admin/analytics/overview?range=7d&timezone=Asia%2FShanghai
```

自定义范围使用：

```http
GET /console/admin/analytics/overview?from=2026-07-01&to=2026-07-10&timezone=Asia%2FShanghai
```

规则：

- `to` 为包含端日期，后端转换成下一个自然日的排他上界。
- 自定义范围不得超过 180 天。
- 无效时区、日期或倒置区间返回稳定的参数错误。
- 查询使用只读事务，响应包含 `generatedAt`、周期边界和时区。

## 响应结构

响应采用面向页面的读模型，避免前端重复解释统计口径：

```json
{
  "generatedAt": "2026-07-11T02:42:00Z",
  "timezone": "Asia/Shanghai",
  "period": {},
  "comparisonPeriod": {},
  "summary": {},
  "headline": {},
  "trend": [],
  "funnel": [],
  "breakdowns": {
    "sources": [],
    "landingPages": [],
    "countries": [],
    "locales": [],
    "deploymentSources": [],
    "failureCodes": [],
    "plans": [],
    "checkoutRegions": [],
    "billingCycles": [],
    "offers": []
  },
  "revenue": {},
  "dataQuality": {},
  "insights": []
}
```

每个指标同时返回当前值、对比值、绝对变化和变化率。分母为零时变化率或转化率为 `null`，不伪造 0%。

## 时间口径

- `today`：上海时区当天零点至当前时刻，对比昨日同时段。
- `7d`：最近七个完整自然日，对比紧邻的前七日。
- `30d`：最近三十个完整自然日，对比紧邻的前三十日。
- `90d`：最近九十个完整自然日，对比紧邻的前九十日。
- 自定义：完整自然日，与同等长度的前置周期比较。
- 趋势粒度：今天按小时；七天和三十天按日；九十天按周；自定义由后端根据跨度选择。

## 唯一用户旅程

登录前使用 `acquisition_id` 识别同一旅程，注册或认领后使用 `user_id`。注册和认领事件负责连接两段身份；无法连接的历史数据进入 `unknown`，不静默删除。

主旅程阶段：

1. **访问**：周期内发生 `landing_view` 的唯一获客标识。
2. **提交发布**：发生 `deploy_submitted` 的唯一旅程。
3. **发布成功**：部署业务事实为成功状态，且能通过 `deployment_id` 或用户关联到旅程。
4. **注册或认领**：周期内完成注册，或游客项目被正式用户认领。
5. **激活**：已登录用户首次成功发布后七日内再次成功发布；只统计完整观察窗口。
6. **发起结账**：存在 `subscription_checkouts` 的唯一用户。
7. **订阅激活**：Stripe Webhook 最终写入激活状态的唯一用户。

每个阶段返回唯一人数、辅助事件数、从上一步转化率、流失人数和同期变化。漏斗不要求所有用户严格按同一页面顺序操作，统计查询通过身份和业务事实建立阶段关系。

## 业务事实优先级

- 发布成功和失败以 `deployments` 为准，`deployment_ready` 与 `deployment_failed` 用于归因连接和幂等核对。
- 注册以用户记录为准，注册事件用于连接获客标识。
- 游客认领以项目从游客项目变为正式归属为准，认领事件用于连接旅程。
- Checkout 以 `subscription_checkouts` 为准。
- 订阅激活以 Stripe Webhook 写入的订阅事实为准。
- 收入以 `subscription_payments` 的实收减退款为准，不使用前端 `purchase` 镜像。
- 不同币种分别展示，不做隐含汇率换算。

## 内部获客与诊断维度

来源按以下优先级归类：明确 UTM 来源、Badge 推荐、外部 referrer 域名、Direct、Unknown。页面展示来源带来的访问人数、发布成功人数、激活人数、订阅人数及各阶段转化率，而不是只展示流量占比。

第一版诊断维度：

- 来源、Campaign 和落地页；
- 国家/地区和界面语言；
- 部署来源：Web、Guest、CLI、MCP、Extension、API、Redeploy；
- 上传或部署失败码；
- Free、月付 Pro、年付 Pro；
- Checkout 地区、账期、Offer 和触发入口；
- Public 与 Password 发布结果。

各维度默认展示前五项和“其他”，支持展开完整列表。小样本行显示样本不足，不进行强结论排序。

## 收入指标

- 周期净收入：周期内实收金额减已记录退款金额，按币种分别展示。
- 新增订阅：周期内首次激活的月付和年付订阅。
- 当前有效订阅：当前状态为 Active 的月付和年付订阅。
- Checkout 转化：发起 Checkout 的唯一用户到订阅激活的唯一用户。
- 取消与 Past Due 作为风险指标展示，但不与当期新增收入混为一谈。
- 续费收入保留在收入趋势中，但不计入新增订阅人数。

## 数据链路完整度

驾驶舱必须显示分析可信范围：

- 成功部署可连接到唯一旅程的比例；
- 注册/认领可连接到原始获客标识的比例；
- Checkout 可连接到入口、Offer 和地区的比例；
- 具有明确来源的访问比例；
- 服务端事实事件与业务事实数量差异。

任何关键关联率低于 90% 时显示警告；低于 70% 时相关经营建议降级或不生成。

## 经营结论与建议规则

建议由后端返回：

```json
{
  "id": "claim-rate-down",
  "severity": "HIGH",
  "title": "优化发布成功页的注册认领入口",
  "evidence": [],
  "impactUsers": 18,
  "action": "检查成功页入口文案、位置和注册返回链路",
  "relatedSection": "funnel.claimed"
}
```

第一版规则覆盖：

- 提交正常但成功率下降：检查失败码、格式和发布服务。
- 发布成功增长但注册/认领率下降：优化成功页认领入口。
- 注册正常但 7 日激活下降：优化再次发布和固定链接更新引导。
- Checkout 增长但订阅激活下降：检查支付失败、价格地区和 Checkout 流失。
- 单一来源流量增长但发布成功率显著低于整体：调整对应落地页或流量投入。
- 特定地区访问增长但激活或付费显著偏低：核对本地化、价格和流量意图。
- 某稳定失败码显著增长：优先修复对应上传体验或后端处理。

规则最低样本量默认为 20 人；没有足够样本时不生成百分比驱动建议。页面最多突出三条建议，按严重程度、影响人数和证据可信度排序。

## 页面信息架构

页面采用克制、编辑化的经营叙事风格，延续 PreviewShip 现有 Geist 字体、颜色变量、圆角和组件语言，不制作与产品割裂的霓虹数据大屏。

阅读顺序：

1. 顶部周期选择、时区、刷新和数据更新时间；
2. 深色经营结论卡与数据完整度状态；
3. 访问、发布成功、激活、新增订阅、净收入五个核心结果；
4. 可交互唯一用户旅程；
5. 30/90 日趋势；
6. 来源、地区、失败原因和商业化诊断；
7. 最值得执行的经营建议；
8. 指标定义和数据说明。

## 交互与无障碍

- 点击或键盘选择漏斗阶段，诊断模块随之显示该阶段的来源、地区和失败原因。
- 所有图表支持鼠标 hover 和键盘 focus 查看精确值。
- 图表同时提供可读摘要；颜色不是唯一信号，配合箭头、符号、标签和数字。
- 日期切换使用 URL 查询参数，刷新和分享页面时保留选择。
- 桌面端使用不超过三列的层级网格；移动端保持单列叙事顺序。
- 图表使用轻量 SVG 和现有组件，不引入大型图表依赖。
- 采用骨架屏、稳定尺寸和渐进呈现，避免加载时布局跳动。
- 空数据、低样本、链路不完整、权限不足和查询失败拥有不同状态。

## 隐私与安全

- API 只返回聚合数据，不返回邮箱、完整 referrer、GCLID、GBRAID、WBRAID、密码、文件名、HTML、Markdown、API Key 或原始 IP。
- 路径和来源在服务端归类后返回，避免将可能包含查询参数的原始值暴露给页面。
- 小样本维度不提供用户明细下钻，降低重新识别风险。
- 原始异常只进入服务端日志，页面展示口语化错误和重试动作。

## 性能与索引

- 自定义查询最大 180 天。
- 聚合在数据库中完成，避免逐事件 Java 处理和 N+1 查询。
- 复用现有 `occurred_at`、事件名、用户、获客、项目和部署索引，并为实际查询计划补充复合索引。
- 趋势和各诊断维度在同一服务调用中生成，前端只发起一个主要请求。
- 实施时使用真实规模的 `EXPLAIN ANALYZE` 或等价测试确认常用七天和三十天查询。

## 异常处理

- 分母为零返回 `null`，前端显示“暂无足够数据”。
- 缺失历史字段归入 Unknown，并计入完整度指标。
- 查询失败返回统一错误结构；不返回 SQL 或原始异常。
- 单一建议规则异常不能影响基础指标响应，记录日志并跳过该建议。
- 页面保留上一份成功数据并标记为旧数据，同时提供重新加载按钮。

## 测试与验收

后端测试覆盖：

- 管理员和非管理员权限；
- 上海自然日、今天同时段、当前和对比周期边界；
- 匿名旅程注册或认领后的连接与去重；
- 游客、登录、重复、失败和多来源部署；
- 7 日激活及未成熟 cohort 排除；
- Checkout、激活、续费、取消、Past Due、退款和多币种；
- Direct、Badge、Unknown、地区、落地页和失败码分类；
- 链路完整度、低样本抑制和建议排序；
- 空数据和历史缺失字段。

前端测试覆盖：

- 日期范围与 URL 参数；
- 数据格式化、百分比和多币种显示；
- 加载、空数据、旧数据、错误和无权限状态；
- 漏斗选择与诊断联动；
- 建议证据跳转；
- 键盘操作和移动端布局。

交付前执行 Backend 测试、Frontend 测试、lint 和生产构建，并使用本地浏览器验证桌面与移动端，保存验收截图。

## 明确不在第一版范围

- 外部 GA、Google Ads、GSC 或 Bing API 实时集成；
- 自由拖拽、自定义指标或 SQL 查询；
- AI 自动生成经营结论；
- 原始用户事件和邮箱级明细浏览；
- 自动修改投放预算、页面或产品配置；
- 汇率换算后的跨币种总收入。
